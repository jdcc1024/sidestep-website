AI Coding Workflow — Module Breakdown

Matt Pocock's methodology separated into 5 core modules. Master each one before setting up your coding environment.

1. LLM Constraints 2. Requirements & Planning 3. Task Architecture 4. Implementation & Agents 5. Codebase Design & QA
1

Understanding LLM Constraints

Foundation

Before building anything, you need to understand how LLMs actually work — their strengths and fundamental limitations. Every decision in this workflow stems from these two constraints.

🧠 Smart Zone vs. Dumb Zone

LLMs operate within a "smart zone" when they start fresh with minimal context. As more tokens accumulate in a session, quality degrades — complexity increases quadratically with each token added (like adding teams to a football league).

Smart Zone

Fresh session, focused task, clear system prompt, minimal prior context. The AI is sharp and produces its best work.

Dumb Zone

Long session, bloated context, multiple unrelated topics discussed. The AI starts forgetting, contradicting itself, and making errors.

Practical rule: Keep every AI task small enough to complete in a fresh or near-fresh context. If a task requires too much context, it's too big — split it.

🎬 The "Memento" Problem

LLMs have no persistent memory between sessions. Like the character in Memento, they wake up fresh every time with no recollection of previous work. Every session follows the same lifecycle:

System Prompt → Exploration → Implementation → Testing → Session Ends
Implication: You can't rely on AI "remembering" decisions from last session. Everything it needs must be written down in documents (PRDs, issues, CLAUDE.md files) that get fed into the next session.

🔄 Clearing vs. Compacting Context

When context gets heavy, you have two options:

Clear Context (Preferred)

Reset completely back to system prompt. AI returns to smart zone. Decisions are preserved in external documents, not conversation history.

Compact Context

Summarize the conversation into fewer tokens. Retains some history but summaries lose nuance. Can still drift into the dumb zone.

Matt's preference: Clear context entirely and rely on well-written documents to carry decisions forward. This keeps the AI consistently operating in its smart zone.

2

Requirements & Planning

Human-in-the-Loop

Planning is ALWAYS human-in-the-loop. This module covers how to go from a vague idea to a structured, AI-ready specification. The goal: reach a "shared understanding" between you and the AI.

🎯 The Design Concept (Shared Understanding)

From Frederick P. Brooks: effective collaboration requires reaching a "design concept" — a shared mental model between all participants. When working with AI, you need the LLM to understand the same destination you have in mind.

How to achieve it: Through structured conversation — not by dumping a spec and hoping it "gets it." The grilling process and PRD creation are tools for building this shared understanding incrementally.

🔥 The "Grill Me" Technique

Before writing any code, have the AI interrogate your idea. It asks hard questions about your requirements — surfacing gaps, edge cases, and assumptions you haven't thought through.

Vague Idea → AI Asks Questions → You Answer & Decide → Shared Understanding

Sessions can involve up to 80 questions. This isn't wasted time — it's the fastest path to alignment. You can also feed in meeting transcripts or domain expert input to enrich the grilling.

Why it works: Prevents the #1 failure mode in AI-assisted development — misalignment. The cost of answering 80 questions upfront is far less than rebuilding a feature that missed the mark.

📄 Product Requirements Document (PRD)

The PRD is your "destination document" — it captures WHERE you're going, not HOW to get there. After grilling, summarize findings into a PRD that will guide all subsequent implementation.

Key components:

Problem Statement → Proposed Solution → User Stories → Implementation Decisions → Testing Strategy
Important distinction: This is NOT "specs to code" (writing a doc and blindly converting it). You still need to understand and manage the code. The PRD guides AI; it doesn't replace your judgment.

🗺️ Two Documents: Destination + Journey

Matt advocates for two complementary documents:

Destination (PRD)

What are we building? For whom? What does success look like? What's out of scope? This document is relatively stable once written.

Journey (Multi-Phase Plan)

How do we get there? What order? What depends on what? This is your Kanban board / issue tracker — it evolves as you learn.

🚫 Anti-Pattern: "Specs to Code"

Matt explicitly critiques the movement that says "just write a specification and AI converts it to code — never look at the code." This fails because:

Why it fails: Understanding and managing code is essential. You need to know what AI produced to review it, test it, debug it, and evolve it. The PRD guides AI — it doesn't replace your responsibility as an engineer.
3

Task Architecture

Structure

How you decompose work determines whether AI can execute it effectively. This module covers the art of breaking a PRD into tasks that stay in the "smart zone" and can run in parallel.

🔪 Vertical Slices (Tracer Bullets)

Instead of building layer-by-layer (all database, then all API, then all frontend), build thin end-to-end features that cut through every layer of the stack.

Horizontal (Don't)

Build all DB schemas → all API routes → all frontend pages. No feedback until the very end. Integration issues discovered late. High risk.

Vertical (Do)

One feature from DB → API → UI. Immediate feedback at each step. You can test and ship incrementally. Low risk.

The analogy: Anti-aircraft gunners used "tracer bullets" — visible rounds that showed where shots were landing, allowing real-time course correction. Vertical slices give you the same visibility into your project's trajectory.

Example: Instead of "build the entire gamification system," your first slice might be: "Award points for lesson completion, visible on a dashboard" — one thin path through schema → service → API → UI.

🔀 Directed Acyclic Graph (DAG) of Tasks

Organize tasks into a DAG — a dependency graph where tasks flow in one direction with no circular dependencies. This enables parallelization.

Phase 1: Foundation → Phase 2: Core Features → Phase 3: Polish

Within each phase, tasks should be independent of each other — multiple agents (or sessions) can work on them simultaneously without conflicts.

Key properties of good tasks: Small (stays in smart zone), independent (no cross-task dependencies within a phase), clearly defined (has acceptance criteria or tests), and vertically sliced (touches all relevant layers).

📋 From PRD to Kanban Board

The transformation pipeline from requirements to actionable work:

PRD → AI Generates Issues → Human Reviews & Adjusts → Kanban Board → Implementation
Human review is essential: Even though AI generates the issues, you must review the plan. AI tends to propose horizontal slices by default — you push for vertical ones. AI may miss business nuances — you add them back in.

⚡ Parallelization Strategy

Tasks within a phase run in parallel rather than sequentially. This is how you multiply throughput when using AI agents.

How it works: If Phase 2 has 5 independent tasks (each a vertical slice), 5 separate AI sessions can execute them simultaneously. Each session starts fresh (smart zone), picks up one issue, implements it with TDD, and produces a PR. Sequential execution limits you to one agent at a time.

🏗️ Owning Your Planning Stack

Matt advocates for owning and customizing your planning process rather than blindly adopting a framework. You need observability — the ability to see what went wrong and adjust.

Principle: Use frameworks as starting points, not gospel. If vertical slices don't work for a particular task, adapt. If the AI's generated plan is wrong, override it. Maintain control over your process.

4

Implementation & Agents

Execution

Once planning is complete, implementation shifts from "human-in-the-loop" to "AFK" (away from keyboard). This module covers how to run AI agents autonomously and how TDD provides the feedback loop they need.

🧑‍💻 Human-in-the-Loop vs. AFK Tasks

Not all work requires your active presence. Categorize every task:

Human-in-the-Loop (Active)

Planning, PRD writing, architecture decisions, UX design, business logic validation, code review. These ALWAYS require you.

AFK (Autonomous)

Implementing well-defined features, writing boilerplate, running test suites, fixing lint errors, data migrations, refactoring. Delegate these.

The "night shift" metaphor: After thorough human planning during the day, AI agents run autonomously like a night shift — executing the well-defined plan without needing supervision.

🧪 Test-Driven Development (TDD)

TDD is the single most important technique for AI-assisted coding. It provides the feedback loop that AI needs to produce correct code.

RED: Write Failing Test → GREEN: Implement Until Pass → REFACTOR: Clean Up
Why TDD + AI is powerful: The test defines "done." AI has a binary signal: does it pass or not? Without this, AI produces code that looks right but may not function correctly. Tests eliminate ambiguity and give the agent a clear objective.

Watch out: AI can sometimes "cheat" tests — writing implementations that technically pass but don't solve the real problem. Using tools like npx vitest and well-designed test cases makes cheating harder.

🔁 Feedback Loops Are Everything

Without feedback loops, AI produces unreliable output. The quality of your feedback loop directly determines the quality of AI's code.

Types of feedback loops:

Test Results Linter Errors Type Checking Build Errors Runtime Errors
Improving feedback loops = improving AI output. If your project has no tests, no type system, and no linter — AI is flying blind. Invest in tooling that gives AI clear pass/fail signals.

🤖 Sub-Agents & Delegation

Sub-agents are separate LLM contexts that handle complex sub-tasks independently. The main agent delegates work, the sub-agent explores and returns a concise summary.

Benefits: Keeps the main agent's context clean (smart zone). Sub-agents can explore complex problems without overwhelming the primary session's token usage. Results are summarized back concisely.

⚙️ The AFK Agent Loop

Matt demonstrates a bash script ("Ralph") that runs Claude autonomously in a loop:

Collect Issues from Backlog → Prioritize (Bugs → Infra → Features) → Agent Picks Task → Implements with TDD → Commits & Moves to Next
Sandboxing: Uses Docker containers to isolate agent work. Each task runs in its own work tree, preventing conflicts. Start sequential to learn the system, then explore parallelization.

📐 Push vs. Pull Information

Two ways to get standards and context to AI agents:

Push (Send Directly)

Instructions baked into system prompt or task description. Good for reviewers who need to enforce specific standards.

Pull (Agent Finds It)

Agent discovers standards by reading config files, CLAUDE.md, or project docs as needed. Good for implementation — agent grabs what's relevant.

Codebase Design & Quality Assurance

Sustainability

The architecture of your codebase determines how effectively AI can work within it. A well-designed codebase amplifies AI; a poorly designed one makes it struggle. QA is where you impose your personal taste.

🏛️ Deep Modules vs. Shallow Modules

From John Ousterhout's "A Philosophy of Software Design" — module structure matters enormously for AI effectiveness.

Shallow Modules (Bad for AI)

Many tiny files with complex interdependencies. Hard to understand in isolation. Difficult to test. AI gets confused navigating them.

Deep Modules (Good for AI)

Simple public interfaces hiding extensive functionality within. Easy to test from the outside. AI can implement changes confidently without understanding every internal detail.

Design principle: Create services with simple APIs but rich internals. E.g., OrderService.createOrder(data) hides all validation, payment, email, and inventory logic behind one clean interface.

🧱 Testable Architecture

Building an easily testable codebase directly improves AI performance — because tests ARE the feedback loop. If your code is hard to test, AI can't get clear signals.

What makes code testable: Clear module boundaries, dependency injection, pure functions where possible, services that can be instantiated with test databases, and well-defined interfaces between layers.

🔍 Three Layers of QA

Quality assurance in AI-assisted development happens at three levels:

1. AI Self-Review → 2. Automated Tests → 3. Human Manual QA
AI Self-Review: Clear context and have a fresh AI session review the code. It's now in the "smart zone" for reviewing, catching issues the implementing agent missed.

Automated Tests: The TDD suite catches regressions and verifies functional correctness continuously.

Human Manual QA: This is where YOU impose taste, judgment, and business knowledge. Don't skip this — it's what separates quality software from generic output.

✋ Manual QA = Imposing Your Taste

Teams that try to fully automate development end up with applications that feel generic and lack quality. Manual QA is your opportunity to inject taste, standards, and craftsmanship.

What to check manually: Does the UX feel right? Does the code read clearly? Is it over-engineered or under-engineered? Does it match the team's style? Would you be proud to ship this? These are human judgment calls that can't be automated.

📏 Small Pull Requests

Keep PRs small and self-contained. Each vertical slice becomes one PR. This reduces review complexity and catches issues faster.

Benefits: Easier to review (fewer loops), easier to revert if something breaks, clear scope per change. Use a separate AI reviewer (e.g., Opus for reviewing, Sonnet for implementing) to get high-quality automated reviews before human review.

🗂️ Documentation That Doesn't Rot

Keep documentation relevant and current. Old PRDs that no longer reflect the code become misleading — Matt calls this "doc rot."

Keep

Living docs that reflect current state: CLAUDE.md, active architecture decisions, current module interfaces, coding standards.

Remove

Outdated PRDs that describe features as-planned (not as-built), obsolete architecture docs, stale README sections that mislead AI.

🧭 Staying Familiar with Your Codebase

As you delegate more to AI, there's a risk of losing familiarity with your own codebase. Matt's solution: focus on designing module interfaces rather than implementing every detail.

Your role shifts to: Defining the "shapes" and behaviors of modules — what they accept, what they return, how they interact. You don't need to co-review every line of implementation, but you must understand the architecture and boundaries.
