AI-Assisted Development Workflow

Key takeaways from Matt Pocock's workshop — applied to building a small business web platform

Core Concept

The "Smart Zone" vs. "Dumb Zone"

LLMs work best at the start of a session with minimal context. As more tokens accumulate, quality degrades — complexity increases quadratically.

Smart Zone

Fresh context, small focused tasks, clear instructions. AI produces high-quality output.

vs

Dumb Zone

Overloaded context, long rambling sessions, vague goals. AI makes mistakes & forgets.

For your platform: Break your build into small, independent tasks — don't try to build everything in one session.

Planning

Start with a Product Requirements Document

Before writing code, define your destination. A PRD captures the shared understanding between you and AI.

1

Problem Statement

What business problem does your platform solve?

2

User Stories

Who uses it and what do they need to accomplish?

3

Implementation Decisions

Tech stack, constraints, and scope boundaries.

4

Testing Strategy

How will you verify it works correctly?

Alignment

Stress-Test Your Requirements

Use AI to challenge your assumptions before building. Matt calls this the "Grill Me" technique — have AI ask you hard questions about your plan.

Why It Matters

Prevents costly misalignment. Catches edge cases, missing features, and unclear requirements early.

How To Do It

Ask AI to question your brief: "What's unclear? What could go wrong? What am I missing?" Get to shared understanding.

For Your Platform

Before building, have AI grill you on: target users, payment flows, auth needs, mobile requirements, integrations.

When To Stop

When you've reached a clear "design concept" — both you and AI understand the same destination.

Architecture

Build in Vertical Slices (Tracer Bullets)

Don't build all database tables, then all APIs, then all frontend. Instead, build thin end-to-end features that touch every layer.

❌ Horizontal (Layer by Layer)

Build all DB → all API → all Frontend. No feedback until the very end. Risky for small businesses.

→

✓ Vertical (Feature by Feature)

One feature end-to-end: DB + API + UI. Immediate feedback. Ship incrementally.

Example: Build "user can sign up and see dashboard" as one slice, not "build all auth" separately from "build all pages."

Task Management

Create Independent, Parallelizable Issues

Break your PRD into a Directed Acyclic Graph (DAG) of tasks — independent units that can be worked on simultaneously.

🎯

Small & Focused

Each task stays in the "smart zone" — one clear objective

🔀

Independent

Tasks can run in parallel without blocking each other

📋

Phased

Group into phases with clear dependencies between groups

For your platform: Phase 1 might be auth + basic layout. Phase 2 adds business features. Phase 3 adds polish & integrations.

Quality

Use Test-Driven Development (TDD)

TDD is the single most effective technique for getting quality code from AI agents. Write the test first, then let AI implement until it passes.

R

Red — Write a Failing Test

Define what "correct" looks like before any code exists.

G

Green — AI Implements Until It Passes

AI has a clear feedback loop: does the test pass? If not, keep trying.

R

Refactor — Clean Up

Once passing, improve code quality while tests protect correctness.

Key insight: Without feedback loops, AI produces unreliable code. Tests ARE the feedback loop.

Codebase Design

Design Deep Modules, Not Shallow Ones

How you structure your codebase directly affects how well AI can work with it. Prefer "deep" modules over "shallow" ones.

Shallow Modules

Many tiny files with complex dependencies between them. Hard for AI to navigate. Hard to test.

vs

Deep Modules

Simple interfaces hiding rich functionality. Easy to test end-to-end. AI thrives here.

For your platform: Build a clean service layer (e.g., OrderService, UserService) with simple public methods hiding complex logic inside.

Workflow

Know When You're Needed (and When You're Not)

Not all tasks require your attention equally. Categorize work to maximize your time.

🧑 Human-in-the-Loop

Planning, design decisions, UX reviews, business logic validation. You MUST be present. Never automate these.

💤 AFK (Away from Keyboard)

Implementing well-defined features, writing tests, fixing lint errors, data migrations. AI runs autonomously.

Planning = Always Human

PRD writing, architecture decisions, and prioritization always need your judgment and domain knowledge.

Implementation = Delegate

Once a task is well-defined with clear tests, hand it off. Review the output, don't watch it run.

Quality Assurance

Manual QA Imposes Your Taste

AI can write code, but only you know what "good" feels like for your business. Manual QA is where you impose personal standards.

🔍

AI Self-Review

Let AI review its own code with fresh context first

🧪

Automated Tests

Tests catch regressions and verify correctness

👁️

Your Review

You check for taste, UX quality, and business fit

Key point: Don't try to automate everything. Teams that over-automate end up with apps that feel generic and lack quality.

Action Plan

Applying This to Your Web Platform

1

Write Your PRD

Define the business problem, target users, and core features. Have AI grill you on gaps.

2

Slice Into Vertical Features

Identify 3-5 end-to-end features to build first. Each should be independently shippable.

3

Build with TDD

Write tests defining behavior, then let AI implement. Keep tasks small and focused.

4

Review & Iterate

QA each slice, create new issues from findings, and repeat the cycle.
